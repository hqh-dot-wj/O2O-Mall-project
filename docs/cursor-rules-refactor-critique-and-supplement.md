# 改造方案批判性分析与补充

> 针对 `cursor-rules-refactor-plan.md` 的**批判性审视**：不合理之处、Cursor 适配盲区、项目结合不足、缺失的端到端示例、Marketplace 整合。

## 一、方案中的不合理与盲区

### 1.1 直接照搬 ECC 的隐患

| 问题                     | 说明                                                                                                                                    |
| ------------------------ | --------------------------------------------------------------------------------------------------------------------------------------- |
| **ECC 面向 Claude Code** | ECC 的 Hooks、Commands、Agents 是为 Claude Code CLI 设计；Cursor 的 Hooks API、Command 触发方式、Agent 定义格式**可能不同**，需逐项验证 |
| **tmux 依赖**            | ECC 的 beforeShellExecution 阻止「非 tmux 下跑 dev server」——**Windows 用户通常不用 tmux**，本项目规则已写明「主要环境为 Windows」      |
| **~/.claude/sessions/**  | ECC 用 `~/.claude/sessions/`；Cursor 可能用 `~/.cursor/` 或项目内 `.cursor/sessions/`，路径需确认                                       |
| **子代理模型路由**       | ECC 可指定「此 Agent 用 Opus」；**Cursor 的 Agent 是否支持 model 字段？** 需查 Cursor 文档                                              |

### 1.2 与项目现状脱节

| 盲区                              | 说明                                                                                                                                                                                          |
| --------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **已有营销配置器**                | 项目已有 `config-operate-drawer.vue`（RuleSchema 动态表单 + Iframe 预览）、`template-operate-drawer.vue`（Schema Builder）——这是**低代码/配置化**的雏形，改造方案未提及如何与这类既有模式结合 |
| **miniapp-client 有独立 .cursor** | `apps/miniapp-client/.cursor/rules/` 下已有 7 条规则（project-overview、uni-app-patterns、api-http-patterns 等），改造方案只提「miniapp/uniapp.mdc」，未说明与**应用级 rules** 的合并/冲突    |
| **Notion MCP 已启用**             | `.cursor/settings.json` 中 `notion-workspace` 已启用，改造方案提到「MCP 可替代」但未分析：Notion 需求文档 → 任务拆解 → 代码生成的**实际串联**                                                 |
| **.kiro/steering 同步**           | rules README 写明「与 .kiro/steering 对齐」，改造后 rules 大改，**steering 是否同步？** 方案未涉及                                                                                            |

### 1.3 模糊需求场景完全缺失

改造方案**没有**回答：当用户说「帮我开发一个低代码平台的功能」时，**具体会怎么跑**？

- 用户输入模糊 → 谁负责澄清？Rules？Commands？Agent？
- 低代码涉及：Schema 设计、动态表单、预览、后端 CRUD —— 应触发哪些 Skills？顺序是什么？
- task-execution-workflow 假设「已有需求文档」；**没有文档时**怎么办？

### 1.4 Cursor Marketplace 未整合

- 项目有 `notion-workspace` 插件（`.cursor/settings.json`）
- 用户提到 `@cursor\marketplace` —— 可能指 Cursor 的 **Rules Marketplace** 或 **Extensions Marketplace**
- 改造方案**未分析**：是否可从 Marketplace 安装现成 Rules/Skills，与项目自定义规则如何配合？

## 三、端到端示例：模糊需求 → 低代码功能

### 3.1 场景

**用户输入**：「帮我开发一个低代码平台的功能，能配置表单、拖拽布局、实时预览。」

### 3.2 当前方案下的「理想」流程（假设）

| 步骤        | 谁负责                       | 实际会发生什么？                                                                                       |
| ----------- | ---------------------------- | ------------------------------------------------------------------------------------------------------ |
| 1. 需求澄清 | ？                           | 无明确入口。Rules 不会主动问「你要做的是表单配置器还是页面搭建器？」                                   |
| 2. 任务拆解 | task-execution Skill         | **前提是「已有需求文档」**；用户没给文档，Skill 无法触发                                               |
| 3. 架构设计 | architecture-\* rules        | globs 匹配 `docs/design/**` 等，**用户还没创建文档**，不触发                                           |
| 4. 后端实现 | backend-p2c Skill            | 需「PRD 结构化」；用户没给 PRD，Skill 的 description 可能匹配不到                                      |
| 5. 前端实现 | admin-web rules、miniapp-d2c | 若用户说「先做管理端」，会匹配 admin-web；但**低代码**涉及 Schema Builder、动态渲染，现有 rules 未覆盖 |
| 6. 联调验证 | run-tests Command            | 用户需主动执行；无自动触发                                                                             |

**结论**：当前方案**没有**「模糊需求 → 澄清 → 拆解 → 实现」的**端到端编排**。多数 Skills/Rules 假设「输入已结构化」。

### 3.3 应补充的「需求澄清」入口

| 补充项                     | 说明                                                                                                                                      |
| -------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **/clarify 或 /plan 命令** | 当用户输入模糊时，**优先触发**：输出「需要你补充的信息」清单（如：目标用户、表单字段来源、是否需要拖拽、预览形态），再进入 task-execution |
| **低代码专用 Skill**       | 项目已有 Schema Builder、RuleSchema 动态表单，可新增 `lowcode-form-builder` Skill：描述「如何基于 RuleSchema 扩展新玩法、如何加预览」     |
| **AGENTS.md 首条**         | 写「若需求模糊，先澄清再实现；可问用户：目标、范围、现有参考」                                                                            |

### 3.4 结合项目既有能力的示例流程

**用户**：「做一个拼团活动的配置页，能选商品、设人数、设时间。」

| 步骤 | 触发     | 动作                                                                                         |
| ---- | -------- | -------------------------------------------------------------------------------------------- |
| 1    | 用户输入 | Agent 识别「营销配置」类需求                                                                 |
| 2    | 匹配     | 参考 `config-operate-drawer.vue`、`template-operate-drawer.vue`、`rule-validator.service.ts` |
| 3    | Skill    | 若有 `marketing-config-pattern` Skill，描述「玩法模板 + RuleSchema + 配置抽屉 + 预览」       |
| 4    | 实现     | 新增 template 或复用现有 template，扩展 ruleSchema.fields                                    |
| 5    | 验证     | run-tests、code-review                                                                       |

**当前缺失**：无 `marketing-config-pattern` 或类似 Skill；rules 未引导 Agent 去读 `config-operate-drawer.vue` 作为参考。

## 八、多场景分析：不同用户输入下的实际行为与缺口

> 不仅「模糊需求」一种，需覆盖：需求文档+部分实现、修 Bug、优化、重构、跨模块、增量完成等。每个场景分析：**当前会怎样**、**缺口在哪**、**应怎样**。

---

### 场景 A：需求文档 + 后端部分完成 + 前端部分完成

**用户输入**：@需求分析文档.md 「按这个文档继续做，后端已经写了一部分，前端也写了一部分。」

| 问题                   | 当前方案下的行为                                                                 | 缺口                                                         | 应有行为                                                                                                  |
| ---------------------- | -------------------------------------------------------------------------------- | ------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------- |
| **会分析文档吗？**     | 会读文档（用户 @ 了），但**不会自动对比代码**                                    | 无「需求 vs 实现」的 gap 分析流程                            | 先做 gap 分析：文档中的功能清单 vs 现有代码，输出「已完成 / 未完成 / 需修改」                             |
| **会改完后端吗？**     | 可能直接按文档写，**覆盖或冲突已有实现**                                         | 无「增量完成」约束，不区分「新增」与「修改」                 | 识别已实现部分，只补缺失；修改时最小改动，不重写                                                          |
| **会去改前端吗？**     | 若用户未明确说「只做后端」，可能**同时动前后端**；若说「只做后端」则可能忽略前端 | 无「按依赖顺序」约束（应先后端 API → generate-types → 前端） | 先确认后端 API 是否就绪；若未就绪，先完成后端再动前端；**不会**在用户说「继续完成」时只做一边、忽略另一边 |
| **会破坏已有实现吗？** | 有风险：大段重写、改接口签名、忽略已有测试                                       | 无「尊重已完成」的显式规则                                   | Rules 中写「部分实现时：只补 gap，不重写；改接口前确认调用方」                                            |

**典型误操作示例**（当前无规则约束时可能发生）：

| 误操作           | 具体表现                                                                      | 后果                                |
| ---------------- | ----------------------------------------------------------------------------- | ----------------------------------- |
| **重写已完成**   | 后端已有 `OrderExportService`，Agent 按文档「新建」一个，或把已有方法整体替换 | 丢失原有逻辑、测试失效              |
| **乱序修改**     | 先改前端加「导出」按钮，再补后端接口；或后端接口未就绪就改前端，导致类型错误  | 前端调不存在的 API，或手写 any 占位 |
| **只做一边**     | 用户说「继续完成」，Agent 只补完后端就停，**不去改前端**；或反过来            | 功能不完整，用户需再次提醒          |
| **改接口不通知** | 后端把 `exportOrders(query)` 改成 `exportOrders(query, format)`，前端未同步   | 编译通过但运行时错                  |
| **忽略已有测试** | 修改已有 Service 时，未跑或未更新对应 spec                                    | 回归 bug                            |

**应补充**：

- **/gap-analysis 命令**：输入「需求文档 + 可选路径」，输出「已完成项、待完成项、冲突项」
- **task-execution 增强**：支持「增量模式」——任务清单中已完成的打勾，只执行未完成
- **AGENTS.md**：写「部分实现时，先 gap 分析，再按依赖顺序（后端→类型→前端）增量完成；说『继续完成』时，前后端都要检查，不遗漏」

### 场景 B：用户只说「继续完成」

**用户输入**：「继续完成」或「接着做」（无 @ 文档）

| 问题                   | 当前行为                                 | 缺口                                            | 应有行为                                                             |
| ---------------------- | ---------------------------------------- | ----------------------------------------------- | -------------------------------------------------------------------- |
| **知道要完成什么吗？** | 依赖上下文；若新会话则**无上下文**       | 无 session 持久化时，Agent 不知道「上次做到哪」 | sessionEnd 保存进度；sessionStart 加载；或用户 @ 上次的 session 摘要 |
| **会从哪开始？**       | 可能乱猜，或从「看起来未完成」的模块下手 | 无「上次断点」记录                              | 上次 session 应记录「当前任务、已完成、下一步」                      |

**应补充**：Phase 5 的 session 持久化，以及「继续完成」时优先读取 session 摘要。

---

### 场景 C：跨模块需求（后端 + 前端）

**用户输入**：「新增订单导出接口，并在管理后台加一个导出按钮。」

| 问题               | 当前行为                                 | 缺口                               | 应有行为                                                                        |
| ------------------ | ---------------------------------------- | ---------------------------------- | ------------------------------------------------------------------------------- |
| **先做哪边？**     | 可能先做前端（写按钮 + mock），或同时做  | 无「后端优先」的显式规则           | 顺序：后端 API → 构建 → generate-types → 前端调用                               |
| **会生成类型吗？** | 若记得 monorepo 规则会，否则可能手写类型 | 依赖 Rules 被加载；glob 可能未命中 | 在 backend rules 或 common 中写「新增 API 后必须 pnpm generate-types」          |
| **会补测试吗？**   | testing.mdc 要求，但可能忽略             | 无强制                             | afterFileEdit 或 pre-commit 已覆盖；可加「新增 API 必须补 spec」的 Command 检查 |

**应补充**：common 或 backend rules 中写「跨模块时：后端 → generate-types → 前端；禁止前端手写与后端重复的类型」。

---

### 场景 D：修 Bug

**用户输入**：@某文件 「这里报错了」或「点击没反应」

| 问题               | 当前行为                         | 缺口                            | 应有行为                                                     |
| ------------------ | -------------------------------- | ------------------------------- | ------------------------------------------------------------ |
| **会跑测试吗？**   | 可能只改代码，不跑               | 无「修 Bug 必跑相关测试」的强制 | /fix 或类似 Command 要求：修复后跑 `pnpm test -- <相关路径>` |
| **会扩大改动吗？** | 可能顺手「优化」周边，引入新问题 | 无「最小修复」原则              | AGENTS.md：修 Bug 时最小改动，不顺手重构                     |
| **会分析根因吗？** | 可能只修表象                     | 无强制                          | 可建议「先复现、再定位、再修」，但不强制                     |

---

### 场景 E：性能优化

**用户输入**：「这段代码性能不好，优化一下」

| 问题                 | 当前行为               | 缺口                           | 应有行为                               |
| -------------------- | ---------------------- | ------------------------------ | -------------------------------------- |
| **会过度优化吗？**   | 可能微优化、牺牲可读性 | 无「优化前先 profiling」的引导 | 建议：先确认瓶颈，再优化；避免过早优化 |
| **会破坏正确性吗？** | 有风险                 | 优化后必须跑测试               | 同场景 D                               |

---

### 场景 F：重构

**用户输入**：「把这个模块重构一下，太乱了」

| 问题               | 当前行为                                    | 缺口               | 应有行为                                  |
| ------------------ | ------------------------------------------- | ------------------ | ----------------------------------------- |
| **会先写测试吗？** | task-execution 说 TDD，但「重构」可能不触发 | 重构前应有测试保护 | /refactor 命令：先确认有测试，再小步重构  |
| **会大爆炸吗？**   | 可能一次改很多文件                          | 无「小步重构」约束 | 建议：每次只改一个层次/一个职责，提交一次 |

---

### 场景 G：按设计稿实现

**用户输入**：@Figma链接 或 @设计图.png 「按这个做一版」

| 问题                    | 当前行为                                                                | 缺口                                  | 应有行为                                                                                    |
| ----------------------- | ----------------------------------------------------------------------- | ------------------------------------- | ------------------------------------------------------------------------------------------- |
| **会走 D2C 吗？**       | miniapp-d2c 的 glob 是 `apps/miniapp-client/**`，用户未改该路径时不触发 | Skills 按路径触发，不是按「用户意图」 | 需用户 @miniapp-d2c 或说「按 D2C 流程」；或 AGENTS.md 写「设计稿→代码时，参考 miniapp-d2c」 |

### 场景 J：编写测试——谁来验证测试本身是对的？

**用户输入**：「给这个 Service 写测试」或「补一下单测」

| 问题                   | 当前行为                    | 缺口                                            | 应有行为                                                              |
| ---------------------- | --------------------------- | ----------------------------------------------- | --------------------------------------------------------------------- |
| **写的测试就对吗？**   | 单 Agent 写测试，**无校验** | 测试可能：断言错误、漏边界、误通过、误失败      | 不能假设「写出来的测试就是对的」                                      |
| **会做红绿对抗吗？**   | **不会**                    | 改造方案无「双 Agent 对抗」设计                 | 红方：尝试找 bug、写应失败的用例；绿方：写应通过的用例；交叉验证      |
| **会开两个子代理吗？** | **不会**                    | Cursor 是否支持多子代理并行？方案未涉及         | 若支持：Agent A 写实现+正向测试，Agent B 写边界/异常用例，互相 review |
| **有变异测试吗？**     | 无                          | 未提及 mutation testing（改代码看测试能否捕获） | 可选：用 Stryker 等做变异测试，验证测试有效性                         |

**典型问题**：

- AI 写的测试「恰好通过」但断言是错的（如 `expect(result).toBeTruthy()` 对任何非空都通过）
- 漏掉边界：`quantity` 为 0、负数、超上限——测试没写，以为覆盖了
- 测试与实现「同源错误」：实现有 bug，测试按错误逻辑写，一起错

**应补充**：

- **/test-with-adversarial 命令**（若 Cursor 支持多 Agent）：绿方写实现+主路径测试，红方写「应失败」的用例 + 边界用例，双方交叉检查
- **测试审查清单**：写测试后自检——是否覆盖 null/0/负数、是否断言具体值而非 toBeTruthy、是否有对应「应失败」用例
- **AGENTS.md**：写「编写测试时，需自检边界与异常路径；关键逻辑建议人工 review 或红绿对抗」

## 九、总结：方案不是「错」，而是「不完整」

- **Rules 精简、Hooks 优先、Skills 迁移**：方向正确，可执行
- **缺的是**：
  1. **Cursor 适配验证**：Hooks/Commands/Agents 的实际行为
  2. **模糊需求入口**：从「用户一句话」到「可执行任务」的桥梁
  3. **项目能力结合**：营销配置器、Schema Builder、Notion 的串联
  4. **Marketplace 整合**：现成插件/Rules 的利用
  5. **端到端示例**：至少一个完整场景（如「低代码配置页」）的 step-by-step

建议在实施 Phase 1 前，先完成：**Cursor 能力验证** + **模糊需求流程设计** + **项目能力盘点**，再推进改造。
