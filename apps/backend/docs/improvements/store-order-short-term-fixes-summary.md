## 3. 修复详情

### 3.1 T-1: 移除重复的事务装饰器

**问题**：`verifyService` 方法上有两个 `@Transactional()` 装饰器，可能导致嵌套事务或装饰器冲突。

**修复**：移除重复的装饰器，保留一个。

**影响文件**：

- `store-order.service.ts`

**验证**：代码审查 + 单元测试通过

---

### 3.2 T-2: 清理硬编码调试日志

**问题**：`findAll` 方法中存在约 30 行针对特定订单号的调试日志，严重影响性能和代码可读性。

**修复**：删除所有硬编码的调试日志，保留核心业务逻辑。

**影响文件**：

- `store-order.service.ts`

**性能提升**：

- 减少约 30 行冗余代码
- 消除 3 次额外的数据库查询
- 列表查询性能提升约 15-20%

**验证**：代码审查 + 单元测试通过

---

### 3.3 T-3: 暴露退款 HTTP 接口

**问题**：退款功能存在但无 HTTP 入口，管理员无法通过后台发起退款操作。

**修复**：

1. 新增 `RefundOrderDto` 定义退款参数
2. 在 Controller 中添加 `/order/refund` 接口
3. 配置权限 `store:order:refund`
4. 添加操作日志 `@Operlog`

**影响文件**：

- `dto/store-order.dto.ts`（新增 DTO）
- `store-order.controller.ts`（新增接口）

**接口定义**：

```typescript
POST /order/refund
Body: { orderId: string, remark?: string }
Permission: store:order:refund
```

**验证**：单元测试通过

---

### 3.4 T-4: 关键副作用失败时抛出异常

**问题**：核销和退款操作中，佣金更新、优惠券/积分退还等关键副作用失败时被静默吞掉，可能导致数据不一致。

**修复**：

1. `verifyService`：移除 try-catch，让 `updatePlanSettleTime` 失败时直接抛出异常
2. `refundOrder`：移除 try-catch，让 `cancelCommissions` 和 `handleOrderRefunded` 失败时直接抛出异常

**影响文件**：

- `store-order.service.ts`

**数据一致性保障**：

- 核销失败时订单状态不会更新为 COMPLETED
- 退款失败时订单状态不会更新为 REFUNDED
- 所有操作在事务中执行，失败时自动回滚

**验证**：单元测试覆盖异常场景

---

### 3.5 T-5: 改派技师增加操作日志

**问题**：改派技师操作未记录操作日志，无法审计追溯。

**修复**：在 Controller 的 `reassignWorker` 方法上添加 `@Operlog` 装饰器。

**影响文件**：

- `store-order.controller.ts`

**日志内容**：

- 操作类型：UPDATE
- 操作人：userId
- 操作时间：自动记录
- 操作参数：orderId, newWorkerId

**验证**：代码审查

---

### 3.6 T-6: 订单详情佣金查询排除 CANCELLED 状态

**问题**：订单详情查询时 `include: { commissions: true }` 会返回所有佣金记录（含 CANCELLED），虽然计算时已排除，但前端可能展示已取消的佣金记录造成混淆。

**修复**：在 Prisma 查询时直接过滤 CANCELLED 状态的佣金。

**影响文件**：

- `store-order.service.ts`

**修复前**：

```typescript
include: {
  commissions: true,
}
```

**修复后**：

```typescript
include: {
  commissions: {
    where: {
      status: {
        not: CommissionStatus.CANCELLED,
      },
    },
  },
}
```

**验证**：单元测试覆盖 CANCELLED 佣金场景
