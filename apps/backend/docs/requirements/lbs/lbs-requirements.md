## 1. 概述

### 1.1 背景

LBS（Location Based Services）是平台的基础能力域，负责：

- 地理围栏匹配（点是否命中站点服务区）
- 行政区划数据管理（省市区街道）
- 站点与租户的地理映射（租户可服务范围）

本次为重分析版本，目标是修正旧文档中已过时结论，并补齐当前代码中的高优先级风险。

### 1.2 目标

1. 明确 LBS 在后端中的能力边界（LBS vs Client Location vs Order Checkout）。
2. 将代码现状中 P0/P1 风险转化为可执行需求与验收项。
3. 统一服务范围判定模型，避免前台匹配与下单校验规则冲突。
4. 建立可观测、可扩展、可运维的 LBS 基线能力。

### 1.3 范围

本文档覆盖：

- `geo`：空间计算与 PostGIS 查询
- `region`：区划数据初始化与查询
- `station`：站点与围栏管理
- `client/location`：C 端租户匹配与附近租户查询
- `client/order`：下单时位置准入校验（集成点）

## 5. 现有功能详述

### 5.1 接口清单

| 接口                              | 方法 | 说明             | 调用方      |
| --------------------------------- | ---- | ---------------- | ----------- |
| `/lbs/region/list`                | GET  | 查询下级行政区划 | 管理端      |
| `/lbs/region/name/:code`          | GET  | 按编码查区划名称 | 管理端      |
| `/lbs/station`                    | POST | 创建站点与围栏   | 管理端      |
| `/lbs/station/list`               | GET  | 查询站点列表     | 管理端      |
| `/lbs/station/check-region`       | GET  | 坐标命中站点查询 | 管理端/调试 |
| `/client/location/match-tenant`   | POST | 按坐标匹配租户   | C 端        |
| `/client/location/nearby-tenants` | GET  | 查询附近租户     | C 端        |

### 5.2 核心方法清单

| 方法                                          | 所在模块        | 说明                   |
| --------------------------------------------- | --------------- | ---------------------- |
| `GeoService.toPolygonWKT`                     | geo             | 围栏点转 WKT           |
| `GeoService.findStationByPoint`               | geo             | PostGIS 点命中围栏查询 |
| `GeoService.calculateDistance`                | geo             | 球面距离计算           |
| `RegionService.seedRegions`                   | region          | 区划初始化             |
| `RegionService.getTree/getChildren`           | region          | 区划查询               |
| `StationService.create`                       | station         | 创建站点与围栏         |
| `StationService.upsertMainStation`            | station         | 主站点覆盖更新         |
| `ClientLocationService.matchTenantByLocation` | client/location | C 端租户匹配           |
| `OrderCheckoutService.checkLocation`          | client/order    | 下单准入校验           |

### 5.3 业务规则详述

1. 围栏坐标统一按 WGS84 经纬度处理。
2. 位置命中优先通过围栏多边形判断，命中后再映射租户。
3. 下单准入当前使用 `tenant.geoConfig` 半径模型，不直接复用围栏模型。
4. 区划数据为空时在模块初始化阶段自动导入 JSON。

## 6. 现有逻辑不足分析

### 6.1 P0 级缺陷

| 编号 | 问题                                                       | 影响                                        |
| ---- | ---------------------------------------------------------- | ------------------------------------------- |
| D-1  | 站点创建围栏坐标转换链路存在数组维度错误风险               | 围栏 WKT 可能错误入库，导致命中结果不可预测 |
| D-2  | `StationRepository` 软删基类与 `SysStation` 模型字段不一致 | 查询/统计条件可能失真，存在运行时风险       |
| D-3  | LBS 管理接口使用 `@Body() any` 且未补齐权限约束            | 输入质量和权限边界风险高                    |

### 6.2 P1 级缺陷

| 编号 | 问题                                      | 影响                           |
| ---- | ----------------------------------------- | ------------------------------ |
| D-4  | 空间命中查询未显式过滤站点状态            | 停用站点可能被命中             |
| D-5  | `sys_geo_fence.geom` 缺少 GIST 空间索引   | 围栏规模增长后查询性能恶化     |
| D-6  | 区划初始化无分布式锁                      | 多实例并发启动可能重复种子写入 |
| D-7  | 区划缓存装饰器在 LBS 服务中可能未真正生效 | 高并发下重复查询 DB            |

### 6.3 P2 级缺陷

| 编号 | 问题                                         | 影响                         |
| ---- | -------------------------------------------- | ---------------------------- |
| D-8  | 缺少经纬度范围、围栏自交、点数下限等完整校验 | 无效几何数据入库             |
| D-9  | 准入模型双轨（围栏 vs 半径）未统一           | 匹配成功但下单失败等体验冲突 |
| D-10 | 围栏类型字符串硬编码（`SERVICE`）            | 扩展多围栏类型困难           |
| D-11 | DTO 声明支持圆形围栏但实现仅支持多边形       | 文档与实现不一致             |

### 6.4 P3 级缺陷

| 编号 | 问题                                     | 影响             |
| ---- | ---------------------------------------- | ---------------- |
| D-12 | 缺少核心指标监控（命中率、延迟、异常率） | 线上问题发现滞后 |
| D-13 | 缺少站点/围栏批量维护能力                | 运维效率低       |
| D-14 | 缺少统一距离单位转换与展示工具           | 上层业务重复实现 |

## 7. 市面主流系统对标

### 7.1 功能对比矩阵

| 能力               | 当前系统 | 美团 | 饿了么 | 京东到家 | 差距 |
| ------------------ | -------- | ---- | ------ | -------- | ---- |
| 围栏命中           | 有       | 有   | 有     | 有       | 持平 |
| 附近租户查询       | 有       | 有   | 有     | 有       | 持平 |
| 统一准入模型       | 弱       | 强   | 强     | 强       | P1   |
| 空间索引与调优体系 | 弱       | 强   | 强     | 强       | P1   |
| 围栏运营工具       | 弱       | 强   | 强     | 中       | P2   |
| 地址解析服务       | 无       | 有   | 有     | 有       | P2   |

### 7.2 差距总结

1. 核心差距不在“有没有接口”，而在“规则是否统一、是否可运维”。
2. 需要优先完成数据正确性和性能基线，再建设高级功能（地理编码、热力图）。

## 11. 非功能需求

### 11.1 性能

- 位置命中接口 P95 小于 500ms（围栏规模 10 万级）。
- 附近租户查询 P95 小于 300ms（50km 半径）。
- 区划树查询命中缓存后 P95 小于 100ms。

### 11.2 可靠性

- 区划初始化需幂等，多实例并发启动不得重复写入脏数据。
- 围栏写入与站点创建必须事务一致，失败可回滚。
- 空间查询失败需返回统一业务错误并记录 traceId。

### 11.3 安全性

- 管理端 LBS 接口必须鉴权并配置权限点。
- 输入坐标与围栏点必须经过白名单校验和范围校验。
- 日志不得打印敏感个人信息，手机号等字段需脱敏。

### 11.4 可维护性

- 明确 LBS 与 client/order 的依赖边界，统一准入服务出口。
- 关键规则沉淀 Process Spec 与 Rule ID 测试映射。
- 变更围栏模型时必须同步更新需求文档、设计文档和任务清单。
