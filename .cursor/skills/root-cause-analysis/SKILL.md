# Root Cause Analysis（根因分析）

当 bug 原因不明确、修复不符合预期、或系统行为异常时使用。

## 方法：5 Whys

从表象出发，连续追问「为什么」，直到找到可操作的根因。

### 步骤

1. **描述表象**：准确描述观察到的错误行为
2. **第一个 Why**：为什么会出现这个表象？
3. **继续追问**：对每个答案继续问为什么，通常 3-5 层
4. **停止条件**：到达可直接修复的代码/配置/数据层面
5. **验证根因**：修复根因后，表象是否消失？

### 示例

```
表象：页面积分显示错误
  ↓ Why 1：API 返回的积分数据不对
  ↓ Why 2：缓存中的积分未更新
  ↓ Why 3：扣减积分时没有失效缓存
  → 根因：account.service.ts 的 deduct 方法缺少缓存失效逻辑
```

### 常见陷阱

- 停在表面：「前端显示错了」→ 不是根因，继续追问
- 跳过验证：找到疑似根因后必须验证，不要猜
- 多因并发：一个表象可能有多个原因，列出所有可能再逐一排除

## Bug 修复流程

禁止：看到错误 → 直接改代码。

```
Bug → Symptom（表象）→ Hypothesis（假设原因，可能多个）→ Validation（验证）→ Fix（修复）
```

### Hypothesis 生成

列出所有可能原因，按概率排序：

- 最近改过的代码
- 数据层（DB/缓存不一致）
- 逻辑层（条件判断、边界）
- 环境层（配置、依赖版本）

### Validation 方法

- 读代码确认逻辑
- 加日志/断点定位
- 写最小复现用例
- 检查数据库/缓存实际数据

## 输出格式

```
表象：[描述]
Why 链：[1] → [2] → [3]
根因：[具体文件/行/逻辑]
验证方式：[如何确认这是根因]
修复方案：[具体修改]
```
