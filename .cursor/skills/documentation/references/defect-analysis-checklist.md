# 缺陷分析前置检查规范

编写「缺陷分析」章节时，**必须先查阅项目实际代码**，禁止凭模块内部代码推断项目全局结构。

## 跨模块文档查阅

1. **同级模块文档**：`docs/requirements/{能力域}/` 下其他子模块
2. **关联模块**：Finance 与 Marketing、Store 与 PMS 等
3. **C 端接口**：`src/module/client/{能力域}/`

## 常见误判

| 误判场景                  | 正确做法                                                 |
| ------------------------- | -------------------------------------------------------- |
| 认为 Finance 缺少积分系统 | 查阅 Marketing 模块，积分在 `marketing/points/` 中实现   |
| 认为 Finance 缺少优惠券   | 查阅 Marketing 模块，优惠券在 `marketing/coupon/` 中实现 |
| 认为模块缺少 C 端接口     | 查阅 `src/module/client/{能力域}/` 目录                  |

## C 端接口检查

分析某能力域是否缺少 C 端接口时，必须先检查 `src/module/client/`。C 端接口统一在 `module/client/{能力域}/` 下。

## 跨模块依赖检查

确认：被引用的 Service/Repository 已通过 Module exports 导出；调用方通过 Module imports 正确引入。

## P0-P3 分级

- **P0**：阻塞性（认证失败、数据丢失、安全漏洞、租户隔离缺失）
- **P1**：高优先级（N+1 查询、功能缺失、God Class）
- **P2**：中优先级（功能不完善、扩展性不足）
- **P3**：低优先级（优化建议、技术债务）

每个缺陷必须包含**现状、影响、建议**。
