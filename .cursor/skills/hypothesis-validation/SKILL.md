# Hypothesis Validation（假设验证）

当做技术决策、选择方案、或需要验证前提是否正确时使用。
核心：避免「逻辑正确但前提错误」。

## 核心问题

每个决策必须回答：

1. **我的假设是什么？**
2. **支持这个假设的证据是什么？**
3. **什么情况会证明我是错的？（反例）**

## 步骤

### 1. 显式化假设

把隐含的前提写出来。大多数错误决策源于未被检验的假设。

```
用户说：「给积分查询加 Redis 缓存」

隐含假设：
- 假设 1：积分查询是性能瓶颈
- 假设 2：瓶颈在 DB 查询而非网络/序列化
- 假设 3：缓存的一致性成本可接受
```

### 2. 寻找证据

对每个假设，找支持或反对的证据：

| 假设         | 证据类型 | 如何获取                 |
| ------------ | -------- | ------------------------ |
| 查询慢       | 性能数据 | 查日志 P99、慢查询       |
| DB 是瓶颈    | 执行计划 | EXPLAIN ANALYZE          |
| 一致性可接受 | 业务规则 | 积分变动频率、用户容忍度 |

证据类型优先级：

- 代码/数据 > 文档 > 经验 > 猜测
- 没有证据的假设 = 猜测，必须标注

### 3. 构造反例

主动想：什么情况下我的方案会失败？

```
假设：加缓存能解决性能问题
反例：如果积分变动频率很高（每秒数十次），缓存命中率极低，
      加缓存反而增加了失效/更新的复杂度，净效果为负。
```

### 4. 决策

- 假设有证据支持 + 反例不成立 → 执行
- 假设无证据 → 先验证再决策
- 反例成立 → 换方案

## 常见需要验证的假设

| 场景       | 常见错误假设 | 应该验证                         |
| ---------- | ------------ | -------------------------------- |
| 加缓存     | DB 查询慢    | 实际 P99 是多少？有没有索引？    |
| 加 ES      | 数据量大     | 多大？当前 DB 查询耗时？         |
| 用策略模式 | 分支会扩展   | 现在几个分支？未来真的会加？     |
| 加消息队列 | 需要异步     | 同步处理的延迟是否真的不可接受？ |
| 微服务拆分 | 模块耦合     | 拆分后的网络/运维成本是否更高？  |

## 输出格式

```
决策：[要做什么]
假设：
  1. [假设] — 证据：[有/无/待验证] — 反例：[什么情况会失败]
  2. ...
结论：[执行 / 先验证 / 换方案]
```
