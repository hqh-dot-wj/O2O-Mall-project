# Impact Analysis（影响分析）

当修改影响范围异常、改一处多处失败、或涉及数据结构变更时使用。

## 步骤

### 1. 直接影响（一阶）

这个修改直接改了什么？

- **数据层**：DB schema、字段类型、索引、缓存 key
- **接口层**：API 入参/出参、DTO/VO 字段
- **逻辑层**：Service 方法签名、业务规则

### 2. 间接影响（二阶）

直接影响会连锁影响什么？

- **上游**：谁调用了被修改的方法/接口？
- **下游**：被修改的方法/接口调用了谁？
- **类型链**：`@libs/common-types` 变更 → 前端类型 → 组件 Props
- **缓存链**：数据变更 → 缓存失效策略 → 读取一致性
- **测试链**：哪些测试用例会受影响？

### 3. 隐性影响（三阶）

不容易发现但可能出问题的：

- **多租户**：修改是否影响租户隔离？
- **并发**：修改是否引入竞态条件？
- **性能**：修改是否改变查询模式（N+1、全表扫描）？
- **前后端契约**：后端 VO 变更是否需要 `pnpm generate-types`？

## 分析工具

### 依赖追踪

```
被修改的文件/方法
  ├─ 被谁 import？（上游调用者）
  ├─ import 了谁？（下游依赖）
  ├─ 对应的 DTO/VO 被哪些接口使用？
  └─ 对应的测试文件在哪？
```

### 影响范围矩阵

| 影响层 | 受影响文件/模块 | 需要同步修改？ | 风险等级 |
| ------ | --------------- | -------------- | -------- |
| 数据层 |                 | 是/否          | 高/中/低 |
| 接口层 |                 | 是/否          | 高/中/低 |
| 前端   |                 | 是/否          | 高/中/低 |
| 测试   |                 | 是/否          | 高/中/低 |

## 输出格式

```
修改点：[文件/方法]
一阶影响：[直接改了什么]
二阶影响：[连锁影响什么]
隐性风险：[可能出问题的点]
需同步修改：[文件列表]
需重新生成类型：是/否
```
