# 任务执行工作流规范

> 当用户提供需求分析文档后，自动触发此工作流

## 触发条件

当用户提供需求分析文档、缺陷分析文档、改进计划文档、或包含待办任务列表的文档时适用。

## 工作流步骤

### 第 0 步：架构审查

用架构决策框架（`architecture-meta-model.md` + `architecture-checklist.md`）审查需求文档：

1. **相态判断**：确认模块当前相态（探索态/成长态/运营态）
2. **四棱镜快速扫描**：业务价值、时间、能力、复杂性
3. **需求补全检查**：模块边界、跨模块连接、抽象层影响、风险信号
4. **审查输出**：✅ 通过 / ⚠️ 补充 / 🛑 重审

### 第 1 步：创建任务清单

文件位置：`apps/{app}/docs/tasks/{模块名}-task-list.md`

```markdown
# {模块名}任务清单

> 来源：{需求文档路径}
> 架构审查：✅ / ⚠️ / 🛑
> 模块相态：{探索态/成长态/运营态}

## 短期任务（1-2周）

- [ ] T-1: 任务名称 (预估工时)

## 中期任务（1-2月）

- [ ] T-4: 任务名称 (预估工时)
```

### 第 2 步：逐个执行任务

每完成一个任务，必须遵循以下标准流程：

#### 2.1 实现功能代码

- 遵循项目规范（backend-nestjs.md / admin-web-frontend.md）
- 使用类型安全、异常处理、事务等最佳实践
- 代码简洁，单函数不超过 80 行
- 探索态代码必须标注 `// EXPLORATION: 此实现为快速验证，PMF 确认后需重构`

#### 2.2 编写单元测试 ⭐ 必须

- 测试文件位置：与源文件同目录，命名为 `*.spec.ts`
- 测试覆盖：正常情况（Happy Path）、边界情况（Edge Cases）、异常情况（Error Cases）
- 测试文件顶部添加 `// @ts-nocheck`

#### 2.3 运行测试

```bash
pnpm --filter backend test -- service-name.spec
pnpm --filter backend test
```

#### 2.4 更新任务清单

将 `- [ ]` 改为 `- [x]`，可选添加完成时间：`- [x] T-1: SKU 创建时传入 costPrice (0.5h) ✅ 2026-02-26`

### 第 3 步：批量完成后总结

当一批任务完成后（如短期任务全部完成），创建总结文档：

**文件位置**（按 `documentation.md §9` 约定）：

- 后端模块 → `apps/backend/docs/improvements/{模块名}-{批次}-summary.md`
- 前端/小程序 → `apps/{app}/docs/improvements/{模块名}-{批次}-summary.md`
- 项目级 → `docs/improvements/{模块名}-{批次}-summary.md`

**内容包含**：已完成任务列表、测试结果统计、规范合规验证、架构审查补充点落实情况、遗留问题、上线后需要监控的风险信号。

## 工作流原则

### ✅ 必须做

1. **架构审查先行**：收到需求文档后，第一件事是用四棱镜审查
2. **任务清单次之**：审查通过后，创建任务清单
3. **测试驱动**：每个任务必须有对应的单元测试
4. **逐个完成**：一次只专注一个任务，完成后再进行下一个
5. **及时标记**：完成任务后立即更新 checkbox
6. **保持简洁**：任务清单只记录任务，不重复需求分析

### ❌ 不要做

1. **不要跳过架构审查**：即使需求文档看起来很完善，也要过一遍四棱镜
2. **不要跳过测试**：即使是简单的修复，也要写测试
3. **不要批量标记**：完成一个标记一个，不要事后补标记
4. **不要在任务清单中写详细描述**：详细内容在需求文档中
5. **不要盲目信任需求文档的优先级**：用业务价值棱镜重新校验

## 工作流全景图

```
需求文档进来
    ↓
第 0 步：架构审查（四棱镜扫描）
    ├── ✅ 通过 → 第 1 步
    ├── ⚠️ 补充 → 标注补充点 → 第 1 步
    └── 🛑 重审 → 输出问题清单 → 与用户确认 → 修正后重新进入第 0 步
    ↓
第 1 步：创建任务清单（含审查结论 + 模块相态）
    ↓
第 2 步：逐个执行任务
    ├── 2.1 实现功能代码
    ├── 2.2 编写单元测试
    ├── 2.3 运行测试
    └── 2.4 更新任务清单
    ↓
第 3 步：批量完成后总结（含风险信号监控项）
```

## 检查清单

**开始执行前**：

- [ ] 已完成架构审查（第 0 步）
- [ ] 审查结论已记录在任务清单头部
- [ ] 已创建任务清单文档，任务按优先级分组，每个任务有预估工时

**完成每个任务后**：

- [ ] 功能代码已实现
- [ ] 单元测试已编写（覆盖正常/边界/异常）
- [ ] 测试已运行并通过
- [ ] 任务清单已更新（checkbox 标记为完成）

**批量任务完成后**：

- [ ] 所有任务的 checkbox 已标记
- [ ] 已创建完成总结文档
- [ ] 架构审查中的补充点已全部落实
- [ ] 上线后监控项已交付给相关 Owner
