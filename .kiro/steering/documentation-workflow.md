# 文档编写工作流程规范

本规范适用于批量编写项目文档的场景，提供完整的工作流程、工具使用策略和效率优化方法。

---

## 1. 工作流程

### 1.1 准备阶段

**确认范围**：

- 明确需要编写文档的模块列表
- 确定文档类型（需求文档、设计文档）
- 确认文档规范和模板
- 评估工作量和时间

**建立标准**：

- 选择一个代表性模块作为第一个
- 详细编写该模块的文档
- 建立文档模板和质量标准
- 确认工作流程和工具使用

**准备工具**：

- 确认可用的工具（readMultipleFiles、fsWrite、fsAppend等）
- 了解工具限制（如fsWrite限制50行）
- 准备批量处理策略

### 1.2 执行阶段

**单个模块编写流程**：

1. **探索模块结构**

   ```typescript
   listDirectory('src/module/xxx');
   ```

   - 了解文件组织
   - 识别关键文件
   - 确认模块边界

2. **批量读取源代码**

   ```typescript
   readMultipleFiles(['controller.ts', 'service.ts', 'repository.ts', 'dto/*.ts', 'module.ts']);
   ```

   - 一次读取多个文件
   - 减少工具调用次数
   - 快速理解模块

3. **分析业务逻辑**
   - 识别核心功能
   - 理解数据流
   - 识别关键流程
   - 发现潜在缺陷

4. **创建需求文档**

   ```typescript
   // 第一步：写入核心内容（<50行）
   fsWrite('requirements.md', coreContent);

   // 第二步：追加详细内容
   fsAppend('requirements.md', detailContent1);
   fsAppend('requirements.md', detailContent2);
   ```

   - 遵循分段写入策略
   - 包含所有必需章节
   - 包含所有必需图表

5. **创建设计文档**

   ```typescript
   // 同样使用分段写入
   fsWrite('design.md', coreContent);
   fsAppend('design.md', detailContent1);
   fsAppend('design.md', detailContent2);
   ```

   - 遵循分段写入策略
   - 包含14个章节
   - 包含所有必需图表

6. **识别缺陷并分级**
   - 对照代码检查
   - 对照规范检查
   - P0-P3优先级分类
   - 每个缺陷包含现状、影响、建议

**批量处理策略**：

- 同类模块一起处理（如system模块的13个子模块）
- 保持稳定节奏（每个模块30-45分钟）
- 每5-10个模块验收一次
- 及时调整方向

### 1.3 验收阶段

**质量检查**：

- 使用文档质量检查清单（见documentation.md §11.2）
- 检查一致性（术语、格式、风格）
- 检查完整性（章节、图表、缺陷分析）
- 检查准确性（数据、逻辑、引用）

**进度报告**：

- 统计完成模块数
- 记录工作量和效率
- 识别问题和改进点
- 更新剩余工作计划

**经验总结**：

- 记录有效的提示词
- 记录遇到的错误
- 总结最佳实践
- 提出改进建议

## 2. 提示词策略

### 2.1 初始提示（第一次）

**详细指令**：

```
根据 module/xxx 文件夹内容，编写需求文档和设计文档
```

**要点**：

- 明确指定模块路径
- 说明文档类型
- 可以补充特殊要求（如语言、重点关注的方面）

**示例**：

```
根据 module/admin/auth 文件夹内容，编写需求文档和设计文档
```

### 2.2 持续工作提示（后续）

**简洁指令**：

```
继续
```

**要点**：

- 极简高效（仅2个字）
- AI能理解上下文
- 保持工作连贯性
- 减少认知负担

**效果**：

- 使用频率：80%+
- 效率最高
- 错误率最低
- 最推荐使用

**变体**：

```
继续，下一个模块
继续处理剩余模块
```

### 2.3 特殊需求提示

**语言切换**：

```
继续，后续使用中文回复
```

**总结请求**：

```
帮我总结已完成的工作和遇到的问题
还有没有未完成的模块？
```

**质量检查**：

```
检查已完成文档的质量
```

**进度查询**：

```
当前进度如何？还剩多少模块？
```

### 2.4 提示词效果对比

| 提示词类型 | 示例             | 效率       | 准确性     | 推荐度     |
| ---------- | ---------------- | ---------- | ---------- | ---------- |
| 极简提示   | "继续"           | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 简短指令   | "继续下一个"     | ⭐⭐⭐⭐   | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐   |
| 详细指令   | "根据xxx编写..." | ⭐⭐⭐     | ⭐⭐⭐⭐⭐ | ⭐⭐⭐     |
| 长篇描述   | 详细要求...      | ⭐⭐       | ⭐⭐⭐⭐   | ⭐⭐       |

**建议**：

- 第一次使用详细指令建立标准

## 3. 工具使用规范

### 3.1 文件读取

**批量读取优先**（减少工具调用次数）：

```typescript
// ✅ 推荐：一次读取多个文件
readMultipleFiles([
  'controller.ts',
  'service.ts',
  'repository.ts',
  'dto/create-dto.ts',
  'dto/list-dto.ts',
  'module.ts',
]);

// ❌ 不推荐：多次单独读取
readFile('controller.ts');
readFile('service.ts');
readFile('repository.ts');
// ... 多次调用
```

**效果对比**：

- 批量读取：1次工具调用
- 单独读取：6次工具调用
- 效率提升：83%

**适用场景**：

- 了解模块结构
- 分析业务逻辑
- 编写文档前的准备

### 3.2 文件写入

**遵守行数限制**（避免写入失败）：

```typescript
// ✅ 推荐：分段写入
fsWrite(path, coreContent); // <50行
fsAppend(path, detailContent1); // 追加
fsAppend(path, detailContent2); // 追加

// ❌ 不推荐：一次写入过多
fsWrite(path, allContent); // >50行，会失败
```

**分段策略**：

1. 第一段：核心章节框架（<50行）
2. 第二段：详细内容1（建议<100行）
3. 第三段：详细内容2（建议<100行）
4. 根据需要继续追加

**检查方法**：

- 写入前估算行数
- 超过50行立即分段
- 追加时也要控制行数

### 3.3 字符串替换

**先读取再替换**（避免替换失败）：

```typescript
// ✅ 推荐：先读取确认
const content = readFile(path);
// 确认oldStr存在且完全匹配
if (content.includes(oldStr)) {
  strReplace(path, oldStr, newStr);
}

// ❌ 不推荐：盲目替换
strReplace(path, oldStr, newStr); // 可能失败
```

**注意事项**：

- oldStr必须完全匹配（包括空格、换行）
- 考虑使用fsAppend代替strReplace
- 替换失败时切换方案

### 3.4 工具选择决策树

```
需要操作文件？
├─ 读取
│  ├─ 单个文件 → readFile
│  └─ 多个文件 → readMultipleFiles ✅
├─ 创建
│  ├─ <50行 → fsWrite
│  └─ >50行 → fsWrite + fsAppend ✅
├─ 追加
│  └─ fsAppend ✅
└─ 修改
   ├─ 精确替换 → strReplace（先读取确认）
   └─ 追加内容 → fsAppend ✅
```

## 5. 常见问题与解决方案

### 5.1 文件写入失败

**问题描述**：

```
<error>Caught error in create mode: aborted</error>
```

**原因分析**：

- fsWrite内容超过50行限制
- 写入过程中超时
- 文件路径不存在

**解决方案**：

1. 检查内容行数（必须<50行）
2. 改用fsWrite + fsAppend分段写入
3. 确认文件路径正确
4. 每次追加控制在合理范围

**预防措施**：

- 写入前估算行数
- 养成分段写入习惯
- 使用模板控制长度

### 5.2 字符串替换失败

**问题描述**：

```
<error>String '...' not found in file</error>
```

**原因分析**：

- oldStr与实际内容不匹配
- 空格、换行等细节不一致
- 文件内容已变更

**解决方案**：

1. 先用readFile读取文件
2. 确认oldStr完全匹配
3. 考虑改用fsAppend追加
4. 避免盲目使用strReplace

**预防措施**：

- 操作前先读取确认
- 使用精确的oldStr
- 优先使用fsAppend

### 5.3 效率低下

**问题描述**：

- 每个模块耗时过长
- 工作进度缓慢
- 质量不稳定

**原因分析**：

- 未建立标准模板
- 重复详细阅读代码
- 提示词过于复杂
- 工具使用不当

**解决方案**：

1. 第一个模块建立标准
2. 后续模块复用结构
3. 使用简洁提示词（"继续"）
4. 批量读取文件
5. 分段写入文档

**效果对比**：

- 优化前：90-120分钟/个
- 优化后：30-45分钟/个
- 效率提升：60-75%

### 5.4 质量不一致

**问题描述**：

- 不同模块风格不同
- 术语使用不统一
- 深度不一致

**原因分析**：

- 未建立统一标准
- 未定期验收
- 缺少质量检查

**解决方案**：

1. 建立文档模板
2. 统一术语表
3. 定期验收质量
4. 使用检查清单
