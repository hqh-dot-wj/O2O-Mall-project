# 需求文档与设计文档规范

生成或编写**需求文档**、**设计文档**时遵循本规范。

## 1. 文档类型

| 文档类型 | 说明                         | 必含内容                     |
| -------- | ---------------------------- | ---------------------------- |
| 需求文档 | 功能目标、用户故事、验收标准 | 用例图、活动图、状态图       |
| 设计文档 | 技术方案、模块划分、数据流   | 类图、时序图、组件图、部署图 |

## 2. 文字描述规范

- 术语统一、主谓宾完整、量化优先、禁止歧义、引用准确

## 3. 文档结构标准

### 需求文档（11 章节）

概述 → 角色与用例 → 业务流程 → 状态说明 → 现有功能详述 → 现有逻辑不足分析 → 市面主流系统对标 → 跨模块缺陷 → 验收标准 → 演进建议与待办 → 非功能需求

### 设计文档（14 章节）

概述 → 架构与模块 → 领域/数据模型 → 核心流程时序 → 状态与流程 → 部署架构 → 缺陷改进方案 → 架构改进方案 → 接口设计 → 数据库设计 → 缓存设计 → 安全设计 → 性能优化 → 测试策略

## 4. 文件命名规范

文件名必须全部使用小写字母，单词之间用连字符 `-` 分隔。

## 5. 文档目录归类规范

| 文档层级 | 位置               | 适用范围           |
| -------- | ------------------ | ------------------ |
| 项目级   | `docs/`            | 通用文档、部署指南 |
| 应用级   | `apps/{app}/docs/` | 特定应用的技术文档 |

后端模块文档路径与 `src/module/` 同构。

## 6. 缺陷分析前置检查

编写缺陷分析时，**必须先查阅项目实际代码**，禁止凭推断。

- 查阅同级模块文档和关联模块文档
- 检查 `src/module/client/{能力域}/` 目录
- 确认跨模块依赖的 Module exports/imports

## 7. 缺陷分析深度要求

每个缺陷必须包含现状、影响、建议。按 P0-P3 分级：

- P0：阻塞性（认证失败、数据丢失、安全漏洞）
- P1：高优先级（N+1 查询、功能缺失、God Class）
- P2：中优先级（功能不完善、扩展性不足）
- P3：低优先级（优化建议、技术债务）

## 8. 大模块文档编写流程

先整体后子模块：

1. 整体架构评估文档 `{module}-overall-analysis.md`
2. 子模块需求文档
3. 子模块设计文档
