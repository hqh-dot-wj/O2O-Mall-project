# 架构决策元模型（Architecture Decision Meta-Model）

> 架构设计和重构时使用。本文件只包含**可复用的原则**，不含项目特定数据。
> 项目当前相态 → 见 `architecture-playbook.md`
> PR 决策模板 → 见 `architecture-checklist.md`

## 0. 触发条件

| 触发条件           | 对应决策级别 |
| ------------------ | ------------ |
| 新增或替换外部依赖 | L2+          |
| 新增跨模块依赖     | L2+          |
| 数据模型变更       | L2+          |
| 新增应用或共享库   | L3+          |
| 技术选型           | L3+          |
| 架构重构           | L3+          |
| 影响多个应用       | L4           |

## 1. 相态感知

### 三种相态

| 相态   | 特征                     | 核心目标                 |
| ------ | ------------------------ | ------------------------ |
| 探索态 | PMF 未验证、用户 < 1000  | 最快速度验证假设         |
| 成长态 | PMF 已验证、用户快速增长 | 速度与可持续性平衡       |
| 运营态 | 产品成熟、用户规模稳定   | 可靠性、效率、长期可维护 |

### 相态如何影响四棱镜权重

| 棱镜         | 探索态       | 成长态          | 运营态     |
| ------------ | ------------ | --------------- | ---------- |
| 时间棱镜     | 几乎只看短期 | 短期与中期并重  | 三维度均衡 |
| 能力棱镜     | 放宽         | 收紧            | 严格执行   |
| 复杂性棱镜   | 容忍         | 偿还+建立抽象层 | 严格控制   |
| 业务价值棱镜 | 最高权重     | 高权重          | 中等权重   |

同一系统的不同模块可能处于不同相态，**按模块判断，不要一刀切**。

## 2. 决策四棱镜

每个架构决策必须同时通过四个棱镜审视。

### 2.1 时间棱镜：半衰期

| 维度     | 问题                   | 判断标准           |
| -------- | ---------------------- | ------------------ |
| 短期收益 | 本季度能解决什么？     | 明确、可量化       |
| 中期成本 | 1 年后维护成本？       | 团队能否独立排障   |
| 长期锁定 | 3 年后能否低成本替换？ | 是否有清晰抽象边界 |

### 2.2 能力棱镜：团队可达边界

| 维度     | 问题                               | 红线             |
| -------- | ---------------------------------- | ---------------- |
| 认知负担 | 从"会用"到"能排查深度故障"要多久？ | > 3 个月则高风险 |
| 单点风险 | 唯一精通者离职怎么办？             | 无预案则禁止引入 |
| 能力拉伸 | 舒适区边缘拉伸 vs 未知跳跃？       | 跳跃则需降级方案 |

### 2.3 复杂性棱镜：注入成本

| 成本类型 | 具体表现                           |
| -------- | ---------------------------------- |
| 显性成本 | 服务器、许可、运维                 |
| 隐性成本 | 编译时间、部署复杂度、新人上手门槛 |
| 传染成本 | 复杂性是否会扩散到其他模块？       |

**元规则**：每引入一个新技术组件，必须同时回答"它的退出成本是多少"。

### 2.4 业务价值棱镜：真实贡献

| 维度       | 问题                        |
| ---------- | --------------------------- |
| 价值链位置 | 核心链路还是辅助链路？      |
| 技术 ROI   | 投入 X 人天，预期带来什么？ |
| 用户感知   | 用户能直接感知改善吗？      |
| 机会成本   | 做这件事 = 不做什么？       |

### 2.5 棱镜冲突裁决规则

| 优先级 | 规则                                       |
| ------ | ------------------------------------------ |
| P0     | 红线一票否决（能力棱镜红线触发则直接否决） |
| P1     | 相态权重优先                               |
| P2     | 可逆性打破平票（退出成本更低的方案优先）   |
| P3     | 降级方案兜底                               |

## 3. 约束发现

在构思方案之前，先穷举约束：时间、人力、技术、兼容、合规、依赖。

- 约束先于方案
- 区分硬约束和软约束
- 隐性约束比显性约束更危险

## 4. 结构元模型

### 4.1 边界先于结构

核心职责在 3 句话内说清，排除项在 5 句话内说清。两个都说不清 → 不要开始写代码。

### 4.2 连接比组件更贵

最小化连接数、稳固必要连接（接口/类型/版本）、可见化连接。

## 5. 认知陷阱

| 陷阱       | 纠正                         |
| ---------- | ---------------------------- |
| 幸存者偏差 | 回到四棱镜评估自己的约束     |
| 阶段错配   | 用相态判断校准               |
| 速度崇拜   | 速度的价值在于缩短反馈环     |
| 完美主义   | 探索态质量标准是"能验证假设" |
| 技术自嗨   | 无法量化业务收益则降低优先级 |
