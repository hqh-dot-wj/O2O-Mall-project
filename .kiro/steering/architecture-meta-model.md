# 架构决策元模型（Architecture Decision Meta-Model）

> 手动引用: `#architecture`。架构设计和重构时使用。

本文件定义架构决策的"元结构"——不是具体的技术规则，而是生成规则的规则。
当涉及技术选型、架构变更、模块拆分、依赖引入等决策时，必须经过此元模型的推导。

## 核心原则：从模式匹配到模式推导

不依赖"我见过类似问题"的经验匹配，而是通过以下元结构推导出当前问题的最优解。

---

## 0. 相态感知（Phase Awareness）——元模型的前置判断

所有决策规则的权重不是固定的，而是随系统所处的"相态"动态调整。
在应用三棱镜之前，必须先判断当前处于哪个相态。

### 0.1 三种相态

| 相态   | 特征                                      | 核心目标                   | 架构的角色                     |
| ------ | ----------------------------------------- | -------------------------- | ------------------------------ |
| 探索态 | PMF 未验证、用户 < 1000、商业模式假设阶段 | 最快速度验证假设           | 不阻碍速度，保持可抛弃性       |
| 成长态 | PMF 已验证、用户快速增长、团队扩张        | 在速度和可持续性之间找平衡 | 建立关键抽象层，管理复杂性增长 |
| 运营态 | 产品成熟、用户规模稳定、收入可预测        | 可靠性、效率、长期可维护   | 系统性管理复杂性，优化长期成本 |

### 0.2 相态如何影响三棱镜权重

| 棱镜       | 探索态权重                         | 成长态权重                         | 运营态权重                     |
| ---------- | ---------------------------------- | ---------------------------------- | ------------------------------ |
| 时间棱镜   | 几乎只看短期（本周能验证什么）     | 短期与中期并重                     | 三个维度均衡                   |
| 能力棱镜   | 放宽（允许单点依赖，团队小可接受） | 收紧（开始消除单点，建立知识冗余） | 严格执行                       |
| 复杂性棱镜 | 容忍高复杂性（可抛弃即可）         | 开始偿还，建立抽象层               | 严格控制，每引入必评估退出成本 |

### 0.3 相态判断的元规则

- 不要把阶段性策略当成普适原则。"快速试错"是探索态的最优策略，不是所有阶段的最优策略。
- 相态转换有滞后性。团队的心智模式往往滞后于产品的实际相态——产品已经进入成长态，团队还在用探索态的方式工作（或反过来）。
- 同一系统的不同模块可能处于不同相态。核心交易链路可能在运营态，而新的 AI 功能模块可能在探索态。按模块判断，不要一刀切。

### 0.4 本项目的相态判断

本项目（monorepo：backend + admin-web + miniapp-client）整体处于成长态：

- 产品方向已确定（多租户 SaaS + 小程序 C 端）
- 核心模块（用户、权限、租户）已稳定，处于运营态
- 新业务模块（营销、金融、门店）处于成长态，部分子功能仍在探索态
- 因此：核心模块严格执行三棱镜，新业务模块允许适度放宽，但必须在代码中标注"探索态代码"（如 `// EXPLORATION: 此实现为快速验证，PMF 确认后需重构`）

## 1. 决策三棱镜（每个架构决策必须同时通过三个棱镜）

### 1.1 时间棱镜：这个决策的"半衰期"是多久？

| 维度     | 问题                   | 判断标准             |
| -------- | ---------------------- | -------------------- |
| 短期收益 | 本季度能解决什么问题？ | 明确、可量化         |
| 中期成本 | 1年后维护成本如何？    | 团队能否独立排障     |
| 长期锁定 | 3年后能否低成本替换？  | 是否有清晰的抽象边界 |

**元规则**：如果一个决策的短期收益无法覆盖其中期成本的 2 倍，拒绝。

### 1.2 能力棱镜：这个决策对团队能力的真实影响

| 维度     | 问题                                           | 红线               |
| -------- | ---------------------------------------------- | ------------------ |
| 认知负担 | 团队从"会用"到"能排查深度故障"需要多久？       | > 3个月则高风险    |
| 单点风险 | 如果唯一精通此技术的人离职？                   | 无预案则禁止引入   |
| 能力拉伸 | 是在舒适区边缘做有限拉伸，还是风险未知的跳跃？ | 跳跃则需要降级方案 |

**元规则**：技术选择必须在团队当前能力的"可达边界"内，而非"理论最优"。

### 1.3 复杂性棱镜：这个决策向系统注入了多少复杂性？

| 成本类型 | 具体表现                           | 评估方式   |
| -------- | ---------------------------------- | ---------- |
| 显性成本 | 服务器、许可、运维                 | 可直接计算 |
| 隐性成本 | 编译时间、部署复杂度、新人上手门槛 | 需要估算   |
| 传染成本 | 这个复杂性是否会扩散到其他模块？   | 依赖分析   |

**元规则**：每引入一个新的技术组件，必须同时回答"它的退出成本是多少"。

### 1.4 约束波动性（AI 时代的元约束）

传统架构假设"约束是相对稳定的"——技术栈不会每月变、团队能力不会突变、业务模型不会颠覆。
AI 时代打破了这个假设：模型能力每几个月跃迁一次，一个新模型的发布可能让你的整个技术方案过时。

**元规则**：

- 对 AI 能力相关的决策，半衰期按月而非按年评估
- 任何依赖特定 AI 模型能力的设计，必须有"模型可插拔"的抽象层（即使当前只用一个模型）
- 区分"稳定约束"（用户需要什么）和"易变约束"（用什么技术实现），架构的锚点应该在稳定约束上

## 2. 结构元模型（系统设计的"生成规则"）

不是"怎么设计这个系统"，而是"好的系统结构必然满足哪些不变量"。

### 2.1 边界先于结构

在设计内部结构之前，先回答："这个模块/服务绝对不负责什么？"

**推导规则**：

- 如果一个模块的"不负责"清单无法在 3 句话内说清，它的边界不够清晰
- 如果两个模块的"负责"清单有交集，必须明确归属或提取为独立服务

**项目实践**：参照 `backend-nestjs.md §14.1` 的模块依赖方向，新增模块前先画出"负责/不负责"边界。

### 2.2 连接比组件更贵

系统中最昂贵的不是组件本身，而是组件之间的连接（依赖、通信、数据流动）。

**推导规则**：

- 最小化连接数：能不依赖就不依赖
- 稳固必要连接：必须存在的连接要有契约（接口、类型、版本）
- 可见化连接：所有跨模块调用必须可追踪

**项目实践**：

- 跨模块通信只通过 Module exports 的 Service 或 EventEmitter2（参照 §14.1）
- 禁止直接 import 其他模块内部文件
- 新增跨模块依赖时，必须在 PR 中说明依赖方向是否符合架构图

### 2.3 抽象层的稳定性

好的抽象隔离变化。判断一个抽象是否稳定的元规则：

| 信号                              | 含义                       |
| --------------------------------- | -------------------------- |
| 抽象层的接口在 3 次业务迭代中未变 | 稳定，继续保持             |
| 抽象层的接口每次迭代都要改        | 抽象错误，需要重新设计     |
| 抽象层被绕过（直接访问底层）      | 抽象不够用，需要扩展或替换 |

## 3. 风险感知元模型（从"经验直觉"到"结构推导"）

不依赖"我见过这个问题"，而是通过结构性指标推导风险。

### 3.1 复杂性上升的结构信号

| 信号                          | 检测方式                          | 含义         |
| ----------------------------- | --------------------------------- | ------------ |
| 模块间依赖数增长              | 依赖图分析                        | 耦合度上升   |
| 单个 Service 行数持续增长     | 代码行数监控（§14.2 阈值 300 行） | 职责膨胀     |
| 构造函数依赖数增长            | 注入数量（§14.2 阈值 6 个）       | 模块边界模糊 |
| 同一个 bug 在不同模块反复出现 | 缺陷分析                          | 缺少统一抽象 |
| 新人上手某模块的时间显著增长  | 团队反馈                          | 认知负担过高 |

### 3.2 性能退化的结构信号

| 信号                         | 检测方式 | 含义                     |
| ---------------------------- | -------- | ------------------------ |
| P95 延迟缓慢上升（非突变）   | 监控趋势 | 数据量增长未被架构消化   |
| 同一张表被越来越多的模块查询 | SQL 审计 | 热点表，需要读模型或缓存 |
| 事务范围持续扩大             | 代码审查 | 业务耦合导致事务膨胀     |

### 3.3 团队能力退化的结构信号

| 信号                             | 检测方式      | 含义                   |
| -------------------------------- | ------------- | ---------------------- |
| 某个模块只有一个人敢改           | PR 审查人分布 | 知识单点               |
| Code Review 中"看不懂"的反馈增多 | Review 记录   | 复杂性超出团队认知     |
| 技术债标记（TECH-DEBT）密度上升  | 代码扫描      | 偿还速度跟不上积累速度 |

## 4. 决策输出模板

涉及架构决策时（技术选型、模块拆分、依赖引入、重大重构），使用以下结构：

```
### 决策：[简述]

**相态判断**：
- 涉及模块所处相态：探索态 / 成长态 / 运营态
- 三棱镜权重调整说明：

**时间棱镜**：
- 短期收益：
- 中期成本：
- 长期锁定风险：
- 半衰期评估：
- 约束波动性：此决策依赖的外部能力（如 AI 模型）是否可能快速变化？

**能力棱镜**：
- 团队掌握周期：
- 单点风险：
- 降级/退出方案：

**复杂性棱镜**：
- 显性成本：
- 隐性成本：
- 传染范围：
- 退出成本：

**结构影响**：
- 新增/变更的模块边界：
- 新增/变更的连接：
- 对现有抽象层的影响：

**风险信号监控**：
- 上线后需要关注的结构信号：
- 触发回滚/调整的阈值：

**可逆性判断**：
- 此决策是单向门还是双向门？
- 如果是双向门，回退成本是多少？
- 如果是单向门，是否已经过充分验证？
```

## 5. 与现有规范的关系

本元模型不替代具体规范，而是生成和校验具体规范的"上层规则"：

| 现有规范         | 元模型如何校验                      |
| ---------------- | ----------------------------------- |
| §14 架构守护规则 | 模块依赖方向 = 连接最小化的具体实例 |
| §15 SLO 规范     | 性能分级 = 时间棱镜的量化落地       |
| §16 技术债管理   | 债务分类 = 复杂性棱镜的具体度量     |
| §17 DX 规范      | 新人上手时间 = 能力棱镜的验证指标   |
| §10 性能规范     | 大表/QPS 分级 = 风险感知的结构信号  |

当现有规范无法覆盖某个新场景时，回到本元模型的相态判断 + 三棱镜进行推导，而非凭经验猜测。

## 6. 常见认知陷阱（Anti-Patterns）

以下是在应用元模型时需要警惕的思维陷阱：

| 陷阱         | 表现                                               | 元模型的纠正                                                                                   |
| ------------ | -------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| 幸存者偏差   | "XX 公司这么做成功了，我们也应该这么做"            | 成功案例的做法不等于因果关系；回到三棱镜评估你自己的约束                                       |
| 阶段错配     | 产品已进入成长态，团队还在用探索态的方式工作       | 用 §0 的相态判断校准，按模块而非全局判断                                                       |
| 反向阶段错配 | 产品还在探索态，却在用运营态的标准做架构           | 同上；探索态的代码允许"可抛弃"，不必满足所有规范                                               |
| 速度崇拜     | 把"快"本身当目标，而非把"快速学习"当目标           | 速度的价值在于缩短反馈环，不在于产出更多代码                                                   |
| 完美主义     | 在探索态追求完美架构，延误验证                     | 探索态的质量标准是"能验证假设"，不是"能承受10倍流量"                                           |
| 教条主义     | 把任何一套方法论（包括本元模型）当成不可质疑的真理 | 元模型本身也需要随实践迭代；如果三棱镜在某个场景下推导出明显不合理的结论，质疑棱镜而非强行执行 |
